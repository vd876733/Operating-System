Process Abstraction — Program vs. ProcessWhat is a process? How is it different from a program?

📄 Program

  • A passive entity — just a file on disk
  • Contains compiled instructions (machine code)
  • Does not consume CPU or memory by itself
  • Stored as an executable file (.exe, .out, .elf)
  • A program can give rise to many processes

⚙️ Process

  • An active entity — program in execution
  • Has its own memory space (stack, heap, code, data)
  • Consumes CPU time, memory, and OS resources
  • Has a unique PID (Process ID) assigned by OS
  • Has a state (running, waiting, ready, etc.)
Simple Analogy: A recipe in a cookbook = Program. A chef actively cooking that recipe in the kitchen = Process. Two chefs can cook the same recipe simultaneously — two processes from one program.

🧠 Process Memory Layout

When a process runs, the OS gives it a virtual address space divided into sections:

Stack (grows downward ↓)
↕ Free Space (grows toward each other)
Heap (grows upward ↑)
BSS / Uninit Data (global vars)
Data Segment (initialized globals)
Text / Code Segment (instructions)
  • Text – the compiled program instructions (read-only)
  • Data – initialized global and static variables
  • Heap – dynamically allocated memory (malloc/new)
  • Stack – local variables, function call frames, return addresses

Process Control Block (PCB)The OS's data structure for tracking each process

📋 What is a PCB?

The Process Control Block (PCB) is a data structure maintained by the OS for every process. It is the OS's "identity card" for a process — it contains ALL information the OS needs to manage and resume a process. Each process has exactly one PCB. PCBs are stored in the OS kernel's memory.

Process Control Block (PCB)
Process ID (PID)
Unique integer ID (e.g., 2345)
Parent PID (PPID)
PID of parent process
Process State
New / Ready / Running / Waiting / Terminated
Program Counter (PC)
Address of next instruction to execute
CPU Registers
All register values (ACC, SP, GP registers…)
CPU Scheduling Info
Priority, scheduling queue pointers
Memory Management
Base/limit registers, page tables
I/O Status
List of open files, I/O devices in use
Accounting Info
CPU time used, wall clock time, limits

📌 Why PCB is Important

  • When OS switches from one process to another (context switch), it saves the current process's state into its PCB
  • Later, OS restores state from PCB to resume the process exactly where it stopped
  • The PCB is what makes multitasking possible
  • PCBs are linked in a process table inside the kernel
Key insight: Without the PCB, the OS would have no idea where a paused process left off or what resources it was using.

Design of System Calls — Invocation & OS HandlingHow user programs request OS services

📞 What is a System Call?

A system call is the mechanism by which a user-space program requests a service from the OS kernel. Since user programs cannot directly access hardware or kernel memory, they use system calls as a controlled gateway.

Examples: read(), write(), open(), fork(), exec(), exit(), malloc() (uses brk/mmap underneath)

🔄 System Call Invocation — Step by Step

① User Program calls a library function like read(fd, buf, n)
↓
② C Library (glibc) places syscall number in register (e.g., RAX=0 for read on Linux x86-64) and arguments in other registers
↓
③ TRAP / INT instruction CPU switches from User Mode → Kernel Mode. Hardware saves user context (PC, registers) onto kernel stack.
↓
④ Kernel Syscall Handler OS looks up syscall number in the syscall table, jumps to the appropriate handler function
↓
⑤ Service is Performed Kernel performs the requested service (reads from disk, sends data to process, etc.)
↓
⑥ Return to User Mode Kernel puts return value in register, restores user context, switches back to User Mode. Program continues.

📂 Categories of System Calls

CategoryExamples

File Managementopen, close, read, write, lseek
Device Managementioctl, read, write (on device files)

Process Control System Callsfork · wait · exec · getpid · getppid and variants

🍴 fork() — Create a New Process

fork() creates a new process (child) that is an exact copy of the calling process (parent). Both processes continue from the same point in the code after fork.

  • The child gets a copy of the parent's memory, file descriptors, and registers
  • Return value: In parent → child's PID. In child → 0. On error → -1
  • The OS uses Copy-on-Write (CoW): memory is shared until one process modifies it
#include <unistd.h> #include <stdio.h> int main() { pid_t pid = fork(); // Create child process if (pid == 0) { // This block runs in CHILD process printf("Child: PID=%d, Parent=%d\n", getpid(), getppid()); } else if (pid > 0) { // This block runs in PARENT process printf("Parent: PID=%d, Child=%d\n", getpid(), pid); } else { perror("fork failed"); } return 0; }

⏳ wait() / waitpid() — Wait for Child to Finish

The parent calls wait() to suspend itself until a child process terminates. This prevents zombie processes (terminated processes whose PCB is still in the table).

  • wait(NULL) – wait for any child
  • waitpid(pid, &status, 0) – wait for a specific child
  • Returns the child's PID; exit status stored in status variable
Zombie Process: A process that has finished execution but its parent hasn't called wait() yet. Its PCB stays in memory. Orphan Process: A process whose parent has died. Adopted by init (PID 1).
pid_t pid = fork(); if (pid == 0) { printf("Child working...\n"); exit(0); // Child exits } else { int status; wait(&status); // Parent waits for child printf("Child done, exit=%d\n", WEXITSTATUS(status)); }

🔁 exec() — Replace Process Image

exec() family replaces the current process's code, data, and stack with a new program. The PID stays the same but the program changes completely. Used together with fork() to run new programs.

  • execl() – argument list
  • execv() – argument vector
  • execvp() – uses PATH to find the program
  • execle() – with environment variables
  • exec() does NOT return on success (old program is gone)
pid_t pid = fork(); if (pid == 0) { // Child: replace itself with /bin/ls execlp("ls", "ls", "-l", NULL); // If exec returns, there was an error: perror("exec failed"); exit(1); } else { wait(NULL); // Parent waits }
fork + exec pattern: This is how shells work! Shell calls fork(), child calls exec() to run your command, parent calls wait() to wait for it.

🪪 getpid() and getppid()

getpid() — returns the PID of the calling process

getppid() — returns the PID of the parent process

printf("My PID: %d\n", getpid()); printf("Parent PID: %d\n", getppid());

  3. Assembly – converts Assembly to machine code object files (.o)

  2. Creates a new process (PCB, PID)
  3. Allocates virtual address space
  4. Loads code and data segments into memory

  6. Sets PC (Program Counter) to main() entry point
  7. Adds process to Ready Queue

OS Mode of Execution — LDE, Interrupts & System CallsHow the OS maintains control while processes run

⚡ Limited Direct Execution (LDE)

The OS wants to run programs as fast as possible — so it lets the CPU run user code directly (no simulation). But this is limited by two problems the OS must solve:

Problem 1 – Restricted Operations: What if the process wants to do something privileged (access disk, send network packet)? → Solution: System Calls via TRAP
Problem 2 – Taking Back CPU: What if a process never yields the CPU voluntarily? → Solution: Timer Interrupts

⏰ Interrupts

An interrupt is a signal to the CPU that something needs immediate attention. It causes the CPU to stop what it's doing, save state, and run an Interrupt Service Routine (ISR).

TypeCauseExample
Hardware InterruptExternal device signals CPUKeyboard press, timer tick, disk I/O complete
Software Interrupt (TRAP)Program executes INT/SYSCALL instructionSystem call from user program
Exception / FaultCPU detects an errorDivide by zero, page fault, invalid memory access
Timer Interrupt: The OS programs a hardware timer (e.g., every 1–10ms). When it fires, CPU jumps to the OS timer handler, which can preempt the current process and run the scheduler. This is how preemptive multitasking works!

🔁 LDE Protocol (Two Phases)

At Boot Time (OS Setup):

  • OS sets up trap table (which code to run for each interrupt/exception type)
  • OS sets up timer (programs hardware timer chip)

At Runtime (Running a Process):

  1. OS creates process entry, allocates memory, sets up user stack
  2. OS restores registers, jumps to user code → CPU is in User Mode
  3. Process runs normally… until:
  4. Syscall TRAP or Timer Interrupt occurs → CPU switches to Kernel Mode
  5. OS handles it → restores state → returns to process

The Context Switch Mechanism & PCB StateHow the OS switches between processes

🔀 What is a Context Switch?

A context switch is the process of saving the state of the currently running process and restoring the state of another process so it can run. It is purely OS overhead — no useful work happens during a context switch.

  • Context = all the information needed to resume a process: CPU registers, PC, stack pointer, memory maps, etc.
  • This information is stored in / restored from the PCB
  • Triggered by: timer interrupt, I/O wait, higher-priority process becoming ready

Context Switch — Step by Step

📦 What the OS Saves / Restores in PCB During Context Switch

Saved to PCB (outgoing process):

  • Program Counter (PC) — next instruction address
  • All CPU general-purpose registers (R0–R15)
  • Stack Pointer (SP)
  • Base/Limit registers (memory bounds)
  • Process state → changes to Ready or Waiting
  • Page table pointer (memory map)

Restored from PCB (incoming process):

  • PC — resumes exactly where it left off
  • All CPU registers restored
  • Stack Pointer restored
  • Memory map loaded
  • Process state → changes to Running
Context Switch Cost: Context switches are pure overhead (no user work is done). Typical cost: ~1–10 microseconds. High context-switch rates reduce overall system performance. This is why scheduling algorithms matter!

Process Scheduling — Types & CriteriaHow the OS decides which process runs next

🎯 What is Process Scheduling?

The OS scheduler decides which process from the Ready Queue gets the CPU next. Since the CPU can run only one process at a time (per core), all others must wait. The scheduler's goal is to maximize efficiency and fairness.

📚 Types of Schedulers

SchedulerAlso CalledJobFrequency
Long-Term Scheduler Job Scheduler Decides which jobs from the job pool are admitted to Ready Queue (controls degree of multiprogramming) Infrequent (minutes)
Short-Term Scheduler CPU Scheduler / Dispatcher Selects which ready process runs next on the CPU Very frequent (milliseconds)
Medium-Term Scheduler Swapper Temporarily removes (swaps out) a process from memory to disk to reduce load, later swaps it back in Moderate

Scheduling AlgorithmsFCFS · SJF · Priority · Round Robin — with examples

① First Come First Served (FCFS) Non-Preemptive

Rule: Process that arrives first gets the CPU first. Uses a simple FIFO queue. The simplest scheduling algorithm.

  • ✅ Simple to implement, fair in order
  • ❌ Convoy Effect – if a long process is at the front, short processes wait a very long time (poor average waiting time)
  • ❌ Not good for interactive systems

Example: Processes arrive at time 0

② Shortest Job First (SJF) Optimal Avg WT

Rule: The process with the shortest burst time is scheduled next. Gives minimum average waiting time for a given set of processes — provably optimal.

  • ✅ Minimizes average waiting time
  • ❌ Starvation – long processes may never run if short ones keep arriving
  • ❌ Cannot predict burst time in practice — OS must estimate using exponential averaging
  • Preemptive SJF = Shortest Remaining Time First (SRTF)

③ Priority Scheduling Preemptive or Non-Preemptive

Rule: Each process has a priority number. CPU is given to the process with the highest priority. (Convention varies: lower number = higher priority in some systems, higher number in others).

  • ✅ Important tasks run first; good for real-time systems
  • ❌ Starvation – low priority processes may wait forever

  • SJF is a special case where priority = predicted burst time

④ Round Robin (RR) Preemptive Used in Time-Sharing

Rule: Each process gets a fixed time quantum (q) of CPU time. After q expires, the process is preempted and added to the end of the Ready Queue. Most widely used algorithm for general-purpose OS.

  • ✅ Fair — every process gets equal CPU time slices
  • ✅ Good response time for interactive processes
  • ✅ No starvation
  • ❌ If quantum too large → degenerates to FCFS
  • ❌ If quantum too small → too many context switches, high overhead
  • ⚡ Typical quantum: 10ms – 100ms

Case Study — Linux SchedulingHow the real Linux kernel schedules processes

🐧 Overview of Linux Scheduler

Linux uses a modular scheduling framework with multiple scheduling classes, each handling different types of processes. The core algorithm since Linux 2.6.23 is the Completely Fair Scheduler (CFS).

🏛️ Linux Scheduling Classes (in priority order)

ClassPolicyForPriority
Stop Class SCHED_STOP Migration threads, highest-priority kernel tasks Highest
Deadline Class SCHED_DEADLINE Real-time tasks with hard deadlines (earliest deadline first) Very High
Real-Time Class SCHED_FIFO, SCHED_RR Real-time processes (multimedia, audio, robotics) High (0–99)
Fair Class (CFS) SCHED_NORMAL, SCHED_BATCH, SCHED_IDLE All normal user processes Normal

⚖️ Completely Fair Scheduler (CFS) — The Core Algorithm

CFS aims to give each process a fair share of CPU time. It simulates an ideal processor where all N processes run simultaneously at 1/N speed.

Key Concepts:

  • Virtual Runtime (vruntime) – tracks how much CPU time a process has received (weighted by priority)
  • Red-Black Tree – CFS stores all runnable processes in a self-balancing tree ordered by vruntime
  • Leftmost node – process with smallest vruntime runs next (it has been most "cheated" of CPU time)
  • nice values – range −20 (highest priority) to +19 (lowest), used as weight

How CFS Works:

  1. Each process has a vruntime counter
  2. When process runs, its vruntime increases (weighted by nice value)
  3. Scheduler always picks the process with lowest vruntime
  4. Low-priority processes' vruntime increases faster → they run less
  5. Result: high-priority processes run more but all make progress
vruntime formula: vruntime += actual_runtime × (1024 / weight). Lower nice value → higher weight → vruntime grows slower → process gets more CPU time.

🕐 Linux Real-Time Scheduling (SCHED_FIFO & SCHED_RR)

SCHED_FIFO:

  • First-In First-Out within each priority level
  • Process runs until it blocks, exits, or is preempted by higher-priority RT process
  • No time quantum — runs until done
  • Used for: audio servers (PulseAudio), kernel RT threads

SCHED_RR:

  • Round Robin within same priority level
  • Has a time quantum (default 100ms)
  • After quantum expires, process goes to end of its priority queue
  • Preempted immediately if higher-priority RT process becomes ready

✅ Exam Quick Revision — Key Points

  • Program vs Process: Program = passive file on disk. Process = active running instance with PID, memory, state.
  • PCB contains: PID, PPID, process state, PC, registers, memory info, I/O info, scheduling info, accounting info.
  • 5 process states: New → Ready ⇌ Running → Terminated; Running → Waiting (I/O) → Ready (I/O done)
  • System call flow: User calls library → library sets register with syscall# → TRAP → kernel mode → syscall table lookup → kernel handles → return to user mode
  • fork(): returns child PID to parent, 0 to child. Creates exact copy of parent process.
  • exec(): replaces current process with new program. Does NOT return on success.
  • wait(): parent blocks until child terminates. Prevents zombie processes.
  • Zombie: finished child, parent hasn't called wait(). Orphan: parent died first, adopted by init.
  • LDE: OS lets process run directly but controls it via TRAP (system calls) and Timer Interrupts (preemption).
  • Context switch: saves PCB of outgoing process, restores PCB of incoming. Pure overhead.
  • FCFS: simple, convoy effect. SJF: optimal avg WT but starvation. Priority: starvation → fix with aging. RR: fair, preemptive, good for interactive.
  • Formulas: TAT = Completion − Arrival. WT = TAT − Burst. Response = First CPU access − Arrival.
  • Linux CFS: virtual runtime (vruntime) tracked per process. Red-black tree. Lowest vruntime runs next. nice −20 to +19.
  • Linux RT: SCHED_FIFO (no quantum) and SCHED_RR (with quantum), priorities 0–99, always beat CFS processes.
