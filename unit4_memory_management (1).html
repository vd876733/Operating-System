01 — MEMORY ACCESS

Address Bus & Memory Access

Every CPU communicates with RAM through a set of parallel signal lines called the address bus, data bus, and control bus. Understanding how these work is fundamental to understanding memory management.

The Memory Bus Architecture

Three buses cooperate for each memory operation:

  • Address Bus – Carries the memory address the CPU wants to read from or write to. It is unidirectional (CPU → Memory). Width determines addressable space: a 32-bit address bus → 232 = 4 GB; a 64-bit bus → 264 bytes (16 exabytes).
  • Data Bus – Carries the actual data being transferred. It is bidirectional. Width (8, 16, 32, 64 bits) determines how much data moves per cycle.
  • Control Bus – Carries signals like READ, WRITE, INTERRUPT, CLOCK, and MEMORY READY to coordinate the operation.
Fig 1.1 — CPU–Memory Bus Architecture

Memory Access Cycle

Each time the CPU reads or writes memory, it follows a precise sequence:

  1. CPU places the target physical address on the address bus.
  2. CPU asserts READ or WRITE on the control bus.
  3. Memory controller decodes the address and selects the correct RAM row/column.
  4. For a READ: RAM drives data onto the data bus; CPU latches it into a register. For a WRITE: CPU drives data onto the bus; RAM stores it.
  5. Memory signals READY; CPU completes the cycle (takes 50–100 ns for DRAM, 1–5 ns for cache).

02 — PROCESS LAYOUT

Memory View of a Process

When an OS loads a program, it provides each process an illusion of owning a large, private, contiguous address space. This space is divided into well-defined segments or regions.

Classical Process Memory Layout

Fig 2.1 — Virtual Address Space of a Process (low → high)

Segment Descriptions

📦 Text (Code) Segment

Contains the compiled machine instructions of the program. Marked read-only and executable to prevent accidental or malicious modification. Shared between multiple instances of the same program (e.g., two bash shells share one copy of the code).

📊 Data Segment

Holds initialized global and static variables. For example, int x = 10; at global scope lives here. It is read-write. Divided into:

  • Initialized data (.data): variables with explicit initial values.
  • BSS (Block Started by Symbol) (.bss): zero-initialized globals. The OS zeroes this region automatically — the binary doesn't need to store these bytes, keeping the executable file small.

🔵 Heap

Region for dynamic memory allocation at runtime. Functions like malloc(), calloc(), new allocate from here. The heap grows upward (toward higher addresses) as more memory is requested. A runtime library (glibc, jemalloc, etc.) manages free-lists within the heap. The OS expands the heap via brk()/sbrk() system calls.

📚 Stack

Stores local variables, function parameters, return addresses, and saved registers. Organized as a LIFO structure. Grows downward (toward lower addresses). Each function call pushes a stack frame; returning pops it. The OS sets a maximum stack size (e.g., 8 MB on Linux by default).

⚠️ Stack vs Heap Growth

Stack grows downward; Heap grows upward. They grow toward each other and a stack overflow or heap overflow occurs when they collide. Modern OS uses guard pages to detect this.

🔒 Kernel Space

The upper portion of the virtual address space is reserved for the OS kernel. On 32-bit Linux, the split is typically 3 GB (user) / 1 GB (kernel). User-space code cannot access kernel addresses — any attempt triggers a protection fault.

03 — REQUIREMENTS

Process Memory Usage Requirements

An operating system must satisfy several key requirements when managing memory for processes:

Relocation
Programs loaded anywhere in RAM
Protection
Processes isolated from each other
Sharing
Shared libraries, copy-on-write
Logical Organisation
Segments reflect program structure
Physical Organisation
Efficient use of RAM and disk

Relocation

When a program is compiled, the linker assigns addresses starting at 0. However, when it is loaded into RAM, another process may already occupy address 0. The OS (or hardware) must relocate all addresses — translating program-relative (logical) addresses to physical RAM addresses. This is done either at load time (slow, static) or at run time using the MMU (preferred).

Protection

Each process must be prevented from reading or writing another process's memory. The CPU enforces this using the Memory Management Unit (MMU), which checks every memory access against permissions set by the OS. A violation raises a segmentation fault (SIGSEGV).

Sharing

Multiple processes may need to share certain memory regions — for example, a shared library (libc.so) loaded once but used by thousands of processes. The OS allows controlled sharing of physical pages while still providing isolated virtual address spaces. Techniques include copy-on-write (COW) after fork().

04 — VIRTUAL MEMORY

Virtual Memory & System Calls

Virtual memory is the central abstraction of memory management. Each process gets its own large virtual address space, independent of physical RAM size. The OS and MMU collaborate to translate virtual addresses to physical addresses transparently.

The Address Space Abstraction

Virtual memory creates the illusion that each process has exclusive access to a vast, contiguous memory space (e.g., 0 to 248 bytes on x86-64). In reality, only actively used pages are backed by physical RAM; the rest can reside on disk or simply be unmapped.

✅ Benefits of Virtual Memory
  • Programs larger than physical RAM can run (using swap).
  • Isolation: one process cannot corrupt another's memory.
  • Simplified linking: each process can use the same virtual addresses.
  • Efficient sharing of code (shared libraries, fork COW).
  • Demand paging: only load pages when actually needed.

Key System Calls

mmap() — Map Memory

Creates a new mapping in the virtual address space of the calling process. Can map a file into memory or create an anonymous mapping (backed by RAM/swap).

void *mmap(void *addr, size_t length, int prot, int flags,
           int fd, off_t offset);

// Example: anonymous private mapping (like malloc internally)
void *mem = mmap(NULL, 4096,
                  PROT_READ | PROT_WRITE,
                  MAP_PRIVATE | MAP_ANONYMOUS,
                  -1, 0);

// Example: map a file into memory
int fd = open("data.bin", O_RDONLY);
void *view = mmap(NULL, file_size, PROT_READ, MAP_SHARED, fd, 0);
prot flagMeaning
PROT_READPages can be read
PROT_WRITEPages can be written
PROT_EXECPages can be executed
PROT_NONEPages cannot be accessed (guard pages)

munmap() — Unmap Memory

Removes a mapping previously created by mmap(), releasing virtual address space. Physical pages are freed (if not shared).

int munmap(void *addr, size_t length);

sbrk() / brk() — Adjust Heap Break

The program break is the address of the first byte past the end of the uninitialized data segment. brk() sets it directly; sbrk() increments it. These are used internally by malloc() for small allocations.

int  brk(void *addr);         // set break to addr
void *sbrk(intptr_t incr);  // increment break by incr bytes

// Grow heap by 4096 bytes:
void *old_break = sbrk(4096);
// Shrink heap:
sbrk(-4096);

mprotect() — Change Permissions

Alters the access protections of a memory region. Used by JIT compilers (mark pages executable after writing code), security mechanisms (make code read-only), etc.

int mprotect(void *addr, size_t len, int prot);

// Mark page as read-only after writing:
mprotect(page, PAGE_SIZE, PROT_READ);
💡 mmap vs sbrk

Modern allocators prefer mmap for large allocations (>128 KB) because the mapping can be independently freed with munmap. sbrk is used for small allocations since the heap can only shrink from the top. sbrk(-n) returns memory only if the top of the heap is free.

05 — TRANSLATION

Address Translation Mechanisms

The CPU generates virtual (logical) addresses. The MMU hardware translates these to physical addresses before placing them on the memory bus. Three mechanisms exist:

1. Static (Base + Limit) Mapping

The simplest scheme. Each process is given a base register (start of its physical allocation) and a limit register (size of allocation). Every virtual address is checked: 0 ≤ VA < limit, then translated as PA = VA + base.

Fig 5.1 — Base & Limit Register Translation

Drawbacks: Entire process must be contiguous in physical memory. No sharing. Internal fragmentation when multiple processes loaded.

2. Segmentation

Covered in detail in Section 6. Each program is divided into named logical segments (code, data, stack); each segment has its own base + limit. More flexible than flat base/limit.

3. Paging

Covered in detail in Section 7. Both virtual and physical address spaces are divided into fixed-size blocks (pages / frames). A page table maps virtual pages to physical frames.

06 — SEGMENTATION

Segmentation

Segmentation divides the virtual address space into variable-length segments that correspond to logical divisions of a program (code, data, heap, stack, etc.). Each segment is a contiguous block with its own base address and length.

Segment Descriptor Table

A virtual address in a segmented system has two parts: <segment_number, offset>. The segment number indexes into a Segment Descriptor Table (maintained by the OS) to get the base address and limit. The physical address = base + offset, checked against limit.

Fig 6.1 — Segmented Address Translation

Advantages & Disadvantages

✅ Advantages
  • Matches programmer's view of a program.
  • Segments can be independently shared/protected.
  • Segments can grow/shrink independently.
  • No internal fragmentation.
❌ Disadvantages
  • External fragmentation — holes between segments in RAM.
  • Compaction needed periodically (expensive).
  • Variable-length segments complicate allocation.
  • Entire segment must be in memory at once.
07 — PAGING

Paging

Paging eliminates external fragmentation by dividing memory into fixed-size blocks. The virtual address space is split into pages; physical memory is split into same-sized frames (or page frames). A page table maps each virtual page to a physical frame.

Page Size & Address Structure

Typical page size: 4 KB (4096 bytes) on x86/ARM. Some systems support huge pages (2 MB, 1 GB) for performance-critical workloads.

A virtual address is split into two fields:

  • VPN (Virtual Page Number): upper bits — index into page table.
  • Offset: lower bits — byte position within the page (12 bits for 4 KB pages).
Fig 7.1 — Paging Address Translation

Page Table Entry (PTE) Structure

Each entry in a page table stores:

FieldBitsPurpose
Physical Frame Number (PFN)20+ bitsWhich physical frame this page maps to
Present (P)1 bitPage is in RAM (1) or swapped out (0)
Read/Write (R/W)1 bitPage is writable
User/Supervisor (U/S)1 bitUser-mode access allowed
Dirty (D)1 bitPage has been modified since loaded
Accessed (A)1 bitPage has been read/written recently
Execute Disable (XD/NX)1 bitPage cannot be executed (prevents code injection)

Translation Lookaside Buffer (TLB)

Walking the page table on every memory access would halve performance (each access needs 2 memory lookups: page table + actual data). The TLB is a small, fast hardware cache inside the MMU that stores recent VPN→PFN translations.

  • TLB hit: translation found in TLB → 1 cycle penalty.
  • TLB miss: MMU walks the page table in memory → several cycles penalty.
  • Typical TLB size: 32–4096 entries. TLB hit rate is usually >99%.
  • On context switch, TLB must be flushed (or tagged with ASIDs to avoid flushing).

Multi-Level Page Tables

For a 64-bit address space, a single-level page table would require terabytes of RAM. Instead, modern OSes use hierarchical page tables. x86-64 uses 4 levels: PML4 → PDPT → PD → PT → Page. Only the portions of the table that are actually used are allocated, saving memory.

📐 x86-64 Four-Level Paging

Virtual address (48-bit): [PML4 9 bits][PDPT 9 bits][PD 9 bits][PT 9 bits][Offset 12 bits]. Each level is a 4 KB table with 512 entries. The CR3 register holds the physical address of the top-level (PML4) table.

08 — PAGE FAULTS & SWAPPING

Page Faults, Sharing, Permissions & Swapping

Page Faults

A page fault is a hardware exception raised by the MMU when a memory access cannot be satisfied by the current page table. The CPU saves its state and jumps to the OS page fault handler.

Types of Page Faults

TypeCauseOS Action
Minor faultPage not mapped but can be satisfied without disk I/O (e.g., demand-zero page, COW page)Allocate/map a frame, update PTE, resume
Major faultPage is on swap disk (Present bit = 0)Load page from disk into a frame, update PTE, resume
Invalid faultAccess to completely unmapped or protected addressSend SIGSEGV to process; kill if unhandled
Fig 8.1 — Page Fault Handling Flow

Demand Paging

Rather than loading the entire program into RAM at startup, the OS only loads a page when it is first accessed. The page table entry for the unloaded page has Present=0; the first access triggers a page fault, which loads the page. This reduces startup time and memory usage for programs with large but rarely-used code/data.

Page Sharing

Multiple processes can map the same physical frame through different virtual addresses. Common uses:

  • Shared libraries: libc.so code pages are shared read-only among thousands of processes.
  • Copy-on-Write (COW): After fork(), parent and child share all pages (marked read-only). When either writes a shared page, a fault occurs, the OS copies the frame, marks both copies writable, and resumes. This makes fork() extremely fast.
  • Shared memory IPC: Processes can share data regions via mmap(MAP_SHARED) or POSIX shared memory (shm_open).

Read/Write Permissions

Each PTE carries R/W, U/S, and NX (no-execute) bits. The MMU enforces these on every access:

  • Writing to a read-only page → fault → SIGSEGV (or COW if it was a shared mapping).
  • User-mode code accessing a kernel page → fault → SIGSEGV.
  • Executing from a no-execute page → fault → SIGSEGV (prevents shellcode injection).
🔒 W^X / DEP

Write XOR Execute (W^X) is a security policy enforced via NX bits: a page that is writable cannot simultaneously be executable. This prevents common exploit techniques like writing shellcode to the stack and executing it. Windows calls this Data Execution Prevention (DEP).

Swapping & Page Replacement

When physical RAM is full and a new page must be brought in, the OS must evict an existing page to disk (swap space on Linux, pagefile.sys on Windows).

Page Replacement Algorithms

AlgorithmPolicyNotes
OPT (Optimal)Evict page used furthest in futureTheoretical minimum faults; impractical (requires future knowledge)
FIFOEvict oldest loaded pageSimple; suffers from Bélády's anomaly
LRUEvict least recently used pageNear-optimal in practice; expensive to implement exactly
Clock (CLOCK)Circular list with reference bitPractical LRU approximation; used by Linux
LFUEvict least frequently usedDoesn't handle phase changes well

Thrashing

Thrashing occurs when the OS spends more time swapping pages in and out than executing processes — because the combined working sets of all processes exceed physical RAM. Detected by high page-fault rate with low CPU utilization. The OS responds by reducing the degree of multiprogramming (suspending some processes).

Process Memory vs OS Memory

The virtual address space is divided between user space and kernel space:

  • User space: code, data, heap, stack of the process. Accessible only in user mode.
  • Kernel space: kernel code, per-process kernel stacks, page tables, device driver buffers. Accessible only in kernel mode.
  • On x86-64 Linux: user space uses virtual addresses 0x0000000000000000 – 0x00007FFFFFFFFFFF (128 TB); kernel space uses 0xFFFF800000000000 – 0xFFFFFFFFFFFFFFFF (128 TB).
09 — BOOKKEEPING

Memory Bookkeeping & Management

The OS must carefully track which physical frames are free, which are allocated to which process, and what properties each mapping has.

Physical Memory Tracking

Free Frame List / Buddy System

The OS maintains a data structure of all free physical frames. When a page is needed, a free frame is popped; when a page is evicted, its frame is returned to the free list. Linux uses the Buddy System — memory is maintained as blocks of power-of-2 sizes. Splitting a large block to satisfy a smaller request and merging freed adjacent (buddy) blocks prevents fragmentation.

🧩 Buddy System Example

Free pool has one 8-frame block. Process requests 3 frames → system uses a 4-frame block (smallest that fits): splits 8 → 4+4; uses one 4-frame block (internal fragmentation = 1 frame). When freed, checks if its buddy is also free; if so, merges back to 8.

Slab Allocator

For kernel objects (task structs, inodes, dentries) that are frequently allocated and freed in fixed sizes, Linux uses the slab allocator. Slabs are pre-allocated pools for specific object types, eliminating per-allocation overhead and reducing fragmentation.

Virtual Memory Area (VMA) Tracking

Each process's virtual address space is described by a set of VMAs (Virtual Memory Areas) — contiguous ranges with uniform permissions and backing. In Linux, each process has a mm_struct containing a list/rbtree of vm_area_struct objects. Each VMA records:

  • Start and end virtual address.
  • Permissions (read, write, execute).
  • Flags (private, shared, anonymous, file-backed).
  • Pointer to backing file (if file-mapped) and offset.
// Inspect a process's VMAs:
cat /proc/<pid>/maps

// Example output:
00400000-00401000 r-xp 00000000 08:01 123456 /usr/bin/myapp
00600000-00601000 r--p 00000000 08:01 123456 /usr/bin/myapp
00601000-00602000 rw-p 00001000 08:01 123456 /usr/bin/myapp
7fff8000-7fffa000 rwxp 00000000 00:00 0      [stack]

Reference Counting & Copy-On-Write

Each physical page has a reference count — how many page table entries point to it. When a page is shared (via fork or shared mapping), its reference count is incremented. A page is only freed when its reference count drops to zero. COW pages have count > 1 and are marked read-only; the first write triggers a fault that creates a private copy.

Reclaim & OOM Killer

When memory is critically low, the Linux kernel's kswapd daemon reclaims pages by writing dirty pages to swap and freeing clean pages. If reclaim fails to free enough memory, the OOM (Out-Of-Memory) Killer selects a process to terminate using a heuristic score (based on memory usage, priority, etc.).

10 — CASE STUDY

Case Study: Linux OS Memory Management

Architecture Overview

Linux uses a 4-level (or 5-level on newer CPUs) hierarchical paging structure. Every process has its own mm_struct which holds pointers to its page directory and VMA list. The kernel itself is mapped into the upper portion of every process's address space for fast system call entry.

Key Components

Physical Memory Zones

Linux divides physical RAM into zones for historical and hardware reasons:

ZoneRange (x86-64)Purpose
ZONE_DMA0 – 16 MBLegacy ISA DMA devices that can only access low memory
ZONE_DMA320 – 4 GB32-bit DMA-capable devices on 64-bit systems
ZONE_NORMALRemaining RAMRegular kernel and user page allocations
ZONE_HIGHMEM(32-bit only)Memory above 896 MB not directly mapped in kernel space

The Buddy Allocator

Linux's physical allocator manages pages in blocks of 20 to 210 pages (1 to 1024 pages = up to 4 MB). Each zone maintains 11 free lists. The alloc_pages() function satisfies allocation requests; __free_pages() returns pages and coalesces buddies.

The Slab/SLUB Allocator

Built on top of the buddy allocator, the SLUB allocator (modern default) provides object caches for frequently allocated kernel structures. Each cache (kmem_cache) is tuned for a specific object size and type, minimizing fragmentation and allocation latency.

Virtual Memory Subsystem

The do_page_fault() handler in Linux processes all page faults. It looks up the faulting address in the VMA list. If found and access is valid, it calls handle_mm_fault() to populate the PTE (allocate a zero page, load from file, or perform COW). If not found, it delivers SIGSEGV.

Swap System

Linux swap can be a swap partition or a swap file. Pages are tracked by swap entries in the PTE (when Present=0, the PTE stores the swap device and offset). kswapd runs in the background and reclaims pages when the number of free pages falls below pages_low. The LRU policy (approximated via active/inactive page lists) guides eviction decisions.

Transparent Huge Pages (THP)

Linux can automatically back anonymous mappings with 2 MB huge pages instead of 4 KB pages. This reduces TLB pressure for memory-intensive workloads. The khugepaged daemon scans for opportunities to coalesce 4 KB pages into huge pages.

🐧 Useful Linux Memory Commands
free -h                  # show total/used/free RAM and swap
cat /proc/meminfo        # detailed kernel memory statistics
cat /proc/<pid>/maps    # VMAs of a process
cat /proc/<pid>/smaps   # detailed memory stats per VMA
vmstat 1                 # virtual memory statistics
pmap <pid>               # process memory map summary
valgrind ./myapp         # detect memory leaks/errors
11 — CASE STUDY

Case Study: Windows OS Memory Management

Architecture Overview

Windows uses a two-level (or four-level on x64) paging architecture managed by the Memory Manager component of the Windows kernel (ntoskrnl.exe). It provides a rich set of memory management APIs and abstractions distinct from POSIX.

Key Components

Virtual Address Space Layout (x64 Windows)

RegionRangeSize
User space0x0000000000000000 – 0x00007FFFFFFFFFFF128 TB
Kernel space0xFFFF080000000000 – 0xFFFFFFFFFFFFFFFF~128 TB

Windows Memory API

Windows exposes memory management through Win32 API functions:

FunctionPurpose
VirtualAlloc()Reserve and/or commit virtual address space (analogous to mmap)
VirtualFree()Release/decommit memory (analogous to munmap)
VirtualProtect()Change page protection flags (analogous to mprotect)
VirtualQuery()Query information about a virtual memory region
MapViewOfFile()Map a file or named shared memory object into address space
HeapAlloc()Allocate from a process heap (analogous to malloc)

Memory States

Windows pages exist in one of three states:

  • Reserved: Address range claimed but no physical storage backing it. Cannot be accessed.
  • Committed: Physical pages (RAM or pagefile) allocated and ready to use.
  • Free: Not reserved or committed; available for allocation.

Page File & Working Set

Windows uses pagefile.sys as its swap space. The working set of a process is the set of pages currently resident in RAM. The Memory Manager trims working sets using a modified LRU clock algorithm. The Modified Page Writer flushes dirty pages to the pagefile so they can be reused.

Section Objects & Memory-Mapped Files

Windows maps files into memory using Section Objects (called file mappings in Win32). A section object backed by a file provides the same functionality as mmap(MAP_SHARED) on Linux. The Cache Manager uses section objects to implement disk caching — file data is cached in the kernel's virtual address space as mapped sections.

ASLR, DEP, and CFG

  • ASLR (Address Space Layout Randomisation): Base addresses of the executable, DLLs, heap, and stack are randomized at load time, making it harder for attackers to predict target addresses.
  • DEP (Data Execution Prevention): Enforces W^X using NX page table bits. Prevents execution of data pages.
  • CFG (Control Flow Guard): Validates indirect function calls/jumps to prevent return-oriented programming (ROP) exploits.
🪟 Useful Windows Memory Tools
  • Task Manager → Details → Memory: Working Set, commit size per process.
  • Process Explorer (Sysinternals): Detailed per-process memory maps, virtual address space viewer.
  • RAMMap (Sysinternals): Physical memory usage by type (process private, mapped files, standby, etc.).
  • WinDbg: Kernel debugger; !vm command dumps memory manager state.
